1. Repositories and Layers

1.1 Functional layer: meta‑vk‑custom

meta‑vk‑custom implements the platform functionality: FIT generation configuration, initramfs, role awareness, OTA update logic, shared environment (shared‑env) artifacts, and U‑Boot boot scripts.

AreaWhat it provides
FIT enablement Machine include that switches kernel output to fitImage and includes initramfs as a separate FIT node
Initramfs Minimal initramfs image + init script that selects the correct rootfs slot and switches to real rootfs
Shared env shared-env.itb FIT container holding boot cycle, rootfs slot state, optional overrides
Boot script boot-ota.scr (compiled from boot-ota.cmd) to compute roles, select slots, and boot the FIT
OTA agents Update agent and post‑boot health checks (role-aware)

1.2 Security overlay: meta‑vk‑secure (recommended as a separate private repo)

meta‑vk‑secure overlays security behavior (via .bbappend and additional recipes), enabling secure boot hardening: FIT signature enforcement, bundle verification tooling, dm‑verity, encryption.

FeatureMechanismNotes
FIT signature enforcement U‑Boot config + embedded public key Prevents booting modified kernel/DTB/initramfs
Shared env signing Sign shared-env.itb Prevents tampering with boot cycle / slot variables
OTA bundle verification Manifest signature + artifact hashes Protects update channel (can be dev‑bypassed using runtime mode)

LUKS data encryption dm‑crypt / cryptsetup (CAAM-ready) Encrypt per‑core data partitions
Commercial delivery recommendation: keep meta‑vk‑secure in a separate private repo to enable access control, separate licensing, reduced audit scope, and independent security patch cadence.

2. FIT Image (What it is, What it contains)

FIT (Flattened Image Tree) is a U‑Boot boot container format. It can package multiple boot artifacts and include cryptographic signatures. For LS1046A, we use a FIT that includes:

  • Kernel (Image for arm64)
  • Device Tree Blob (DTB) (fsl-ls1046a-frwy.dtb)
  • Initramfs (core-image-minimal-initramfs.cpio.gz) as a separate FIT node

2.1 FIT image conceptual content

fitImage (FIT container)
├─ images/
│  ├─ kernel@1   : Linux kernel Image (arm64)
│  ├─ fdt@1      : Board DTB (fsl-ls1046a-frwy.dtb)
│  └─ ramdisk@1  : initramfs (cpio.gz)
└─ configurations/
   └─ conf@1: kernel + fdt (+ ramdisk)

2.2 Load addresses (typical LS1046A)

Exact addresses depend on memory map and BSP expectations. Typical values used in the configuration: 

Kernel entry/load: 0x80080000
DTB load:          0x90000000
Initramfs load:    0xA0000000

2.3 Why initramfs as a separate FIT node?

  • U‑Boot loads kernel/DTB/ramdisk explicitly from a single artifact (FIT)
  • Signatures can cover all boot components
  • Initramfs can implement early boot logic (choose correct rootfs slot, gather core identity)

3. Storage / Partition Model

This solution uses a combination of:

  • Shared storage (visible to all cores) for rootfs A/B + shared env
  • Per‑core dedicated storage for FIT A/B, local U‑Boot env, and per‑core data

3.2 Storage diagrams

SHARED FLASH (eMMC/SD)

+--------------------------------------------------------------+

+-------------------------------------------------------------------+

+-------------------------------------------------------------------+
Key property: a rootfs update is performed once (Master), but FIT updates are done per-core and can be staged independently.

4. Multi‑Core Architecture

All cores run the same software stack, but their responsibilities can differ per boot depending on role. Role is derived from a shared boot cycle counter plus each core’s identity.

4.1 Core identity inputs

  • vk.core_id — integer (1..3)
  • vk.boot_cycle — shared or local counter used to rotate roles
  • vk.core_role — computed role passed to Linux userspace
  • vk.fit_slot — active per-core FIT slot (A/B)
  • vk.rootfs_slot — active shared rootfs slot (A/B)

4.2 Userspace consumption

A systemd oneshot service reads /proc/cmdline and writes a file such as: 

/run/core-role.env
CORE_ID=1
CORE_ROLE=master
BOOT_CYCLE=0
FIT_SLOT=A
ROOTFS_SLOT=A

5. Role Rotation (“Boot Shift”) Mechanism

5.1 Rotation formula

ROLES = [Master, Slave, Backup]
index = (boot_cycle + core_id - 1) % 3
role  = ROLES[index]

5.2 Example table

boot_cycleCore 1Core 2Core 3
0MasterSlaveBackup
1BackupMasterSlave
2SlaveBackupMaster

5.3 Shared env missing: local fallback

If shared env can’t be read, each core uses a local counter (local_boot_cycle) that it increments. This keeps the shift mechanism working under degraded conditions.

Deterministic shift avoids hardcoding a “forever master”. Over time, each core becomes master.

6. Boot Flow (U‑Boot → Linux)

6.1 U‑Boot responsibilities

  • Load and (optionally) verify shared-env.itb
  • Compute core_role from boot_cycle + core_id
  • Select per‑core FIT slot (A/B), handle rollback counters
  • Select rootfs slot (A/B) based on shared env
  • Boot FIT and pass vk.* parameters via kernel bootargs

6.2 Bootargs example

vk.core_id=1
vk.core_role=master
vk.boot_cycle=0
vk.fit_slot=A
vk.rootfs_slot=A
root=LABEL=rootfsA

6.3 Linux initramfs responsibilities

  • Mount proc, sys, dev
  • Read /proc/cmdline
  • Find correct rootfs partition (LABEL=rootfsA/B)
  • Mount rootfs and switch_root into real system

7. OTA Update Flow (Rootfs A/B + FIT A/B)

7.1 Two independent update domains

DomainScopeUpdated byTypical artifact
Rootfs A/BSharedMaster onlycore-image-ota-*.wic.bz2 or ext4
FIT A/BPer coreEach core independentlyfitImage / ITB

7.2 Rootfs update (Master)

Master writes the new rootfs image to the standby partition (A→B or B→A), then updates shared env to indicate the new active slot on next boot. 

1) Read current shared slot (ota_rootfs_slot=A/B)
2) Choose standby slot
3) dd new rootfs image -> standby partition
4) Update shared env:
   ota_rootfs_slot=
   ota_rootfs_pending=1
   ota_rootfs_confirmed=0
5) Reboot
6) Post-boot health-check confirms and clears pending flag

7.3 FIT update (each core)

Each core stages a new FIT into its local standby slot and sets local env flags for rollback / confirmation. 

1) Read current FIT slot (fit_slot=A/B) from per-core env
2) Choose standby slot
3) flashcp new FIT -> standby area
4) Set flags:
   fit_upgrade_available=1
   fit_boot_count=0
5) Reboot
6) Post-boot health-check confirms:
   fit_boot_confirmed=1
   fit_upgrade_available=0

7.4 Rollback concept

If a core fails to boot the new FIT repeatedly, U‑Boot flips back to the previous slot (A/B) based on a boot counter limit.

Important: Confirmation is core-local for FIT, and shared for rootfs (Master confirms shared rootfs).

8. Security Layer (meta‑vk‑secure)

8.1 Secure vs non‑secure modes

AspectNon‑secure buildSecure build (meta‑vk‑secure)
FIT boot FIT can be unsigned (depending on U‑Boot config) FIT signatures enforced in U‑Boot (reject tampered/unsigned)
Shared env Can be unsigned Signed FIT recommended to prevent tampering with slot/role parameters
OTA bundle verification Optional / not present Manifest signature + hashes (runtime mode controls enforcement)
Rootfs integrity Standard ext4 Optional dm‑verity + read-only rootfs
Data at rest Plain ext4 Optional LUKS/dm‑crypt for per‑core data partitions

8.2 Key management (high-level)

  • FIT signing key: signs FIT images; public key embedded in U‑Boot DTB
  • OTA signing key: signs manifest for OTA bundles; public cert deployed to target
  • SRK (Super Root Keys): used by NXP chain-of-trust tooling; fused into device for production secure boot
Never commit private keys to git. Use offline/HSM storage. Only public certificates should be shipped to the target.

9. “Bypass fuses write” — What is actually bypassable

Critical clarification: once fuses are programmed to enable secure boot in hardware, you cannot bypass ROM-level secure boot checks by software. “Bypass” can only apply to software-level enforcement that happens after U‑Boot is running.

9.1 Not bypassable (hardware enforced)

  • Boot ROM chain-of-trust after fuses are set (ISBC/ESBC enforcement)
  • If fused, unsigned bootloader/images will not run regardless of scripts or flags

9.2 Bypassable (software enforced)

  • OTA bundle verification in userspace (manifest + hashes)
  • Some runtime strictness choices in Linux (e.g., warn vs enforce on verification checks)

9.3 Requested behavior

Signatures remain required for FIT boot (strict), but OTA bundle verification can be bypassed in development using VK_OTA_BUNDLE_VERIFY_MODE.

10. VK_OTA_BUNDLE_VERIFY_MODE (enforce / warn / off)

10.1 What it controls

This flag controls how the script ota-verify-bundle reacts to verification failures. It does not change U‑Boot FIT signature enforcement.

10.2 Configuration file

/etc/vk-secure.conf
VK_OTA_BUNDLE_VERIFY_MODE=enforce|warn|off

10.3 Mode semantics

ModeBehavior on failureExit codeUse case
enforceBlock update1Production
warnLog warning, allow update0Development/integration
offSkip verification entirely0Development only (last resort)
Recommended dev mode: warn (still performs checks and surfaces issues, but does not block iteration).

11. Operational Notes and Recommendations

11.1 Best practices

  • Keep FIT signature enforcement enabled as early as possible in development to match production behavior.
  • Use VK_OTA_BUNDLE_VERIFY_MODE=warn only for development and CI bring-up.
  • In production, set VK_OTA_BUNDLE_VERIFY_MODE=enforce and lock down U‑Boot console if required.
  • Ensure the rootfs update process includes confirmation gating (pending/confirmed) to prevent “stuck in pending” states.
  • Do not overwrite per‑core data partitions during OTA. Keep them as separate WKS images or separate flash regions.

11.2 Common failure modes (and mitigations)
